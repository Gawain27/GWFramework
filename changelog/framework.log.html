0.0.17 P ‚ÄúRuntime World Pipeline & Model Packaging‚Äù ‚Äì released 16 February 2026

üß© Deterministic Runtime World State (Beans)

  • World manager bean: centralised access to the active WorldDef and its lifecycle, with deterministic lookup by id and stable, explicit setters for runtime switching.
  • Section/active-section tracking: structured APIs to enumerate, resolve and update the current section where the player is interacting, including section-local coordinates mapped into world space.
  • Section query utilities: predictable helpers to compute section bounds and world-space transforms for placements, preparing runtime systems (spawn, triggers, navigation) to reason about the world consistently.

üßç Dynamic Entities: Repository & Placement

  • Dynamic entity repository: a dedicated runtime registry that stores, identifies and enumerates dynamic entities (characters, scene objects, temporary effects) independently from static map placements.
  • World/section placement model: entities keep explicit placement metadata: world id, section id/index, and local/world position, with safe APIs to move entities and rebind them across sections.
  • Runtime-friendly identity: entity ids are stable keys suitable for persistence, networking or editor hooks, without coupling to rendering internals.

üé• Camera Management Bean (Focus & Follow)

  • Camera configuration API: a bean to set camera pose (position, yaw/pitch/roll or equivalent) and ‚Äúlook-at‚Äù targets in world or section space.
  • Follow / focus support: camera can lock onto a dynamic entity as a focus target, updating each frame while still allowing optional smoothing / offsets for gameplay feel.
  • Decoupled from rendering: camera state is owned by the camera bean; render compilation reads a snapshot each frame, keeping thread boundaries clean.

üß± Static World Rendering: Render Chunks & Draw Commands

  • RenderChunks strategy (multi-threaded): worker threads compile visible static content into immutable draw command buffers, allowing heavy sorting, visibility filtering and placement-to-quad expansion off the GL thread.
  • Overridable visibility policy: the strategy that selects what to render is pluggable; the default policy renders a 9√ó9 section window with the active section centered.
  • Queue-based handoff: prepared chunks/frames are sent to the render queue as render events, ensuring the GL thread only consumes ready-to-draw data.

üßµ GL-Thread Execution & Texture-Region Support

  • GL-thread command executor: render commands are executed only on the LibGDX render thread, preserving correctness while keeping compilation parallel.
  • Asset-backed region resolution: the asset manager gained a structured way to resolve texture regions directly (including from IAssetPath), enabling template-style sub-rect rendering without reloading textures.
  • Region caching & reuse: region lookups are cached per (texture path + region key) so repeated placements and animated frames avoid allocation churn.

üì¶ Distribution Build: ConfigSet-Scoped Model Packaging

  • Model copy task: Gradle now copies JSON model data for the active configSet projects into the final distribution.
  • Packaged layout: model/<subproj>/{maps,templates,worlds} is merged into ../data/model/{maps,templates,worlds} under the generated distro directory.
  • Runtime alignment: engine pathing resolves assets from ../data/assets and models from ../data/model, matching the packaged structure for both IDE and JAR deployments.

‚úÖ Outcome

  • Static world rendering is now a two-stage pipeline: multi-threaded chunk compilation ‚Üí GL-thread execution, with a clean queue boundary.
  • World and section state are accessible and modifiable through deterministic beans, enabling runtime spawning and gameplay systems.
  • Dynamic entities have a first-class repository and placement model, and the camera can follow them without coupling systems together.
  • Final builds ship with the correct configSet-scoped model data, so maps/templates/worlds load reliably from the packaged distribution.
0.0.16 O ‚ÄúWorld Sections, Gates & Overlays‚Äù ‚Äì released 19 November 2025

üß≠ World Sections Canvas & Layout

  • World sections layout: central WorldIsoRenderer canvas, left section list, and right gate sidebar integrated into a single editor view.
  • Resizable sidebars: left and right panes are hosted in a SplitPane, allowing horizontal drag to resize each sidebar relative to the world view.
  • Scrollable sidebars: both side panels (plane sidebar and world gate sidebar) are ScrollPane-backed, so large maps with many sections or gates remain usable without clipping.

üßä 3D Iso Renderer: Sections, Ghosts & Selection

  • Section-aware world rendering: WorldIsoRenderer now draws every SectionPlacement as a box in world space using a shared yaw/pitch/roll camera, reusing the same projection helpers as the plane view.
  • Rotated selection outline: the yellow selection frame is computed from the section‚Äôs rotated 3D bounding box, so it follows rotXDeg, rotYDeg and rotZDeg instead of assuming axis-aligned sections.
  • Ghost sections while dragging: moving a section uses a dedicated GhostSection model ‚Äì content is drawn with reduced alpha plus a translucent bounding box, making it obvious where the section will land.
  • Depth-stable placement drawing: all plane placements inside sections are collected into DrawItem entries and sorted by camera-space depth, ensuring consistent back-to-front compositing in the iso view.

üéõÔ∏è Per-Axis Translation & Rotation Controls

  • Section transform controls (UI side, wired to the renderer): buttons for +X, -X, +Y, -Y, +Z, -Z move the selected section in world space.
  • Step size input: a numeric field defines the translation step in tiles/units; all six translation buttons use this value, allowing fine or coarse repositioning without re-entering coordinates.
  • Per-axis rotation buttons: additional +X/-X, +Y/-Y, +Z/-Z rotation controls adjust rotXDeg, rotYDeg, rotZDeg with a user-defined degree step.
  • UI state and the yellow world-space selection outline remain in sync as sections are moved and rotated.

üö™ World Gate Sidebar & Linking

  • World-level gate sidebar (WorldGateSidebarPane): new right-hand scrollable panel that operates on WorldDef instead of a single plane.
  • Per-section gate listing: lists all gates belonging to the currently selected section across all planes of its MapDef, using GateEndpoint to identify section + plane + gate.
  • Gate renaming: editing a gate name writes back to the underlying Plane2DMap.GateMeta, and refreshes both the local gate list and the global search index.
  • World gate links: displays all WorldDef.GateLink instances that involve any local gate, with add/remove/rename operations hooked into the world model.
  • Gate search & linking: a searchable combo lists all other gates in the world (excluding the current section); selecting one and hitting ‚ÄúAdd Link‚Äù creates a symmetrical link between the two endpoints if not already present.
  • Duplicate protection: links are compared via a stable endpoint key so adding the same pair twice is silently ignored.

üé® Collision & Gate Overlays

  • Two new overlays toggled from the right sidebar: ‚ÄúShow collision tiles‚Äù and ‚ÄúShow gate tiles‚Äù.
  • Collision overlay: for each placement‚Äôs TemplateDef, solid tiles are rendered in yellow using CollisionShapes strategies, respecting shape and orientation (full rect, half-rect, triangle varieties).
  • Gate overlay: gate tiles (td.gate = true) are drawn as green rectangles on top of the texture, using the same tile grid derived from the placement‚Äôs source region.
  • Both overlays are drawn in the placement‚Äôs local image space via the same affine transform used to project the texture, so they stay perfectly aligned with rotations, scaling and tilt.

üåà Gate & Link Highlighting in the World View

  • Blue gate highlight: selecting a gate in the world gate sidebar calls back into WorldIsoRenderer via setHighlightedGate. The renderer:
    • matches the selected GateEndpoint against each placement (section index + plane key + GateRef.pid()),
    • uses TemplateGateUtils.computeGateIslands() to find the island for the gate index, and
    • draws that island‚Äôs tiles in blue over the placement texture.
  • Purple link highlight: selecting a world gate link passes the GateLink to the renderer via setHighlightedLink; both endpoints are resolved and their gate islands are overdrawn in purple so the connection is visible in context.
  • Overlay-friendly ordering: collision (yellow) and gate (green) overlays are drawn first; blue and purple highlights are rendered last, guaranteeing that selections remain clearly visible.

‚úÖ Outcome

  • World sections can be positioned and rotated directly in a 3D-ish iso view with clear, rotation-aware selection feedback.
  • Gate metadata is now editable and linkable at the world level, not just per-plane.
  • Collision and gate tiles are inspectable in-context, with precise overlays that track texture transforms and tilt.
  • Selected gates and gate links are visually highlighted inside the world, making cross-section navigation and debugging substantially easier.
0.0.15 N ‚ÄúPlanes, Layers & Config Sets‚Äù ‚Äì released 15 November 2025

üß± Plane Editor: Layers, Selection & Tilt

  • Layer-aware selection in PlaneCanvasPane: current layer is always initialised to 0, and selecting a placement updates the active layer in both the canvas and PlaneSidebarPane.
  • Active-layer filter: when a layer is chosen, hit-testing only selects placements on that layer, preventing accidental edits across layers.
  • Smart hit-test override: if you click a placement on a different layer, it is still selected and the active layer automatically switches to that placement‚Äôs layer.
  • Tilt controls per placement in the sidebar: four mutually exclusive modes (Forward, Sideways, Oblique 1, Oblique 2) with a degree spinner (0‚Äì360¬∞). Tilt state is reflected in the instance header line.
  • tiltMode and tiltDegrees are fully wired to the plane view and 3D renderer, with safe clamping and UI sync when changing selection.

üß≠ 3D Iso Renderer: Tilt & Layer Pixel Offsets

  • True 3D-ish placement drawing in IsoRenderer: each plane placement is mapped to world-space quads with orthographic yaw/pitch/roll camera.
  • Tilted placements: tiltMode uses plane-relative axes (u, v, and two obliques) and rotates the placement quad around its center via an axis-angle transform.
  • Layer-based screen offset: each placement is shifted in screen space along the plane‚Äôs fixed axis projection (Z‚Üí(0,0,1), X‚Üí(1,0,0), Y‚Üí(0,1,0)) by layerIndex √ó 1px, letting higher layers visually ‚Äúfloat‚Äù above lower ones.
  • Invertible offsets: invertLayerOffset on Plane2DMap.Placement flips the direction of the per-layer offset, allowing ‚Äúbackwards‚Äù stacking for negative parallax effects.
  • Depth-stable drawing: placements are sorted back-to-front using camera-space depth computed from the rotated Z row of the view matrix, ensuring correct overdraw when planes intersect visually.

üß© Plane Visibility & Grid Toggle

  • Grid toggle refinement: disabling the grid no longer hides the current plane; instead, the selected plane is always rendered as a single highlighted quad.
  • With grid on, behaviour is unchanged: per-tile quads for all planes (or just the selected plane in tile mode) with plane highlighting.
  • With grid off, only the selected plane is drawn, still highlighted, giving a clean view of placements without losing spatial orientation.

üñºÔ∏è Template Gallery: Smarter Thumbnails

  • Adaptive card thumbnails in TemplateGalleryPane: buildCard() no longer assumes a fixed thumbnail height.
  • Thumbnail ImageView fits to the configured width while preserving aspect ratio, respecting the actual template height to avoid cards that are half empty for short assets.
  • Animated templates still show looping thumbs with per-frame snapshots, but sizing is now visually consistent across simple, complex and animated templates.

üìÅ Config-Set‚ÄìScoped Map & Template Repositories

  • MapRepository and TemplateRepository have been reworked to respect the root configSet selected in build.gradle.
  • Both repositories now store JSON under model/maps/{configSet} and model/templates/{configSet} respectively, keeping assets neatly partitioned per game/config environment.
  • Gradle launcher tasks (MapEditor and TemplateEditor) run the JavaFX tools with the correct working directory so their repositories resolve to the configSet-specific paths automatically.
  • Existing JSON loading and listing logic is preserved, but now scoped to the active configuration, avoiding accidental cross-game contamination.

‚úÖ Outcome

  • Plane editing is layer-aware and tilt-capable, with intuitive selection rules.
  • The 3D renderer aligns with plane semantics, supporting tilt, layer depth offsets and inversion in a stable, depth-sorted view.
  • Template gallery cards make better use of space thanks to height-aware thumbnails.
  • Maps and templates are cleanly isolated per configSet, matching the Gradle configuration model.
